# 대리점 관리 시스템 진화 설계안 (초안)

**문서 성격:** 설계 제안 / 논의용 초안 (개발 착수 전)
**작성일:** 2026-06-30
**전제(사용자 의향):** DB 기반으로 전환, 모바일에서도 사용, 향후 "대리점 활동 관리 시스템"으로 확장, 엑셀 업로드 + 웹 편집 모두 단계적 제공

---

## 1. 배경과 목표

현재는 단일 HTML(`agency_map.html`)에 데이터가 빌드 시점에 박혀 있는 정적 구조다. 앞으로는 (1) 사용자가 직접 대리점을 추가·수정하고, (2) 변경이 영구 저장되며, (3) 웹·모바일 어디서든 접근하고, (4) 나중에 방문·상담 등 **활동 이력까지 관리**하는 시스템으로 키운다.

이를 위해 데이터를 코드에서 분리하고, **DB + API(백엔드) + 클라이언트(웹/모바일)** 구조로 전환한다. 핵심은 "지도 보기 앱"에서 "데이터가 살아있는 관리 시스템"으로의 전환이다.

## 2. 목표 아키텍처

```
[웹/모바일 클라이언트]  ──(HTTPS/API)──>  [백엔드 + 인증]  ──>  [데이터베이스]
  · 지도/목록 조회                          · 권한(로그인)         · 대리점 master
  · 대리점 추가/수정(웹)                     · 검증/업서트          · 활동 이력(확장)
  · 엑셀 일괄 업로드                         · 지오코딩 1회 저장     · 사용자/권한
  · (확장) 활동 입력                                                
        │
        └─ Naver Maps(지도·지오코딩)
```

- **데이터는 DB가 단일 소스(SSOT)**. 좌표(위/경도)는 생성·수정 시 1회 지오코딩해 DB에 저장 → 매번 재조회 불필요, 속도↑·API 호출↓.
- **클라이언트는 API만 호출**. 정적 JSON 공개 대신 인증된 API로 접근(PII 보호).
- 현재 지도 앱은 데이터 소스만 "JSON 주입 → API"로 바꾸면 그대로 재사용 가능.

## 3. 기술 선택 권고

**권고: Supabase(관리형 PostgreSQL + 자동 API + 인증) 기반.**

| 항목 | 권고 | 이유 |
|---|---|---|
| 백엔드/DB | **Supabase**(또는 Firebase) | DB+인증+API+스토리지를 즉시 제공, 서버 직접 운영 불필요, 무료 티어로 시작 가능, 모바일/웹 공용 API |
| DB 유형 | **관계형(PostgreSQL)** | 대리점-활동-담당자 관계 및 리포트/집계에 유리(NoSQL보다 통계·조인 강점) |
| 인증/권한 | Supabase Auth + Row Level Security | 로그인 사용자만 조회·편집, 역할별 권한(보기/편집) |
| 프런트(모바일) | **반응형 웹 + PWA** | 별도 앱 개발 없이 폰에서 설치·사용. 필요 시 추후 네이티브 |
| 배포(호스팅) | Vercel/Netlify/Cloudflare Pages(무료) | localhost → 실제 도메인으로 공개해야 모바일 접근 가능 |
| 지도 | 현행 Naver Maps 유지 | 배포 도메인을 NCP Web 서비스 URL에 등록 필요(SalesRouteFinder 키 권장) |

대안: 자체 서버(FastAPI/Node + PostgreSQL)는 통제력이 높지만 설치·운영·인증을 직접 만들어야 함 → 초기 속도와 유지보수 면에서 Supabase 권장.

## 4. 데이터 모델 (초안)

**agencies (대리점 master)** — 현재 데이터 + 안정 키
- id(PK), 대리점코드, 본부코드, 지점코드(고유), 대리점명, 본부명, 지점명, 위촉설계사수, 연락처, 주소(원본), 위도, 경도, 지오코딩상태, 개시일자, 비고, 수정일시

> 지금 슬림 데이터엔 **지점코드/대리점코드가 빠져 있음** → 업서트·중복방지의 기준 키이므로 반드시 복원.

**(확장) activities (활동 이력)** — 활동관리 단계에서 추가
- id, 대리점_id(FK), 담당자_id(FK), 유형(방문/상담/교육 등), 일시, 메모, 결과/후속, 첨부

**(확장) users (담당자/사용자)** — id, 이름, 역할(관리자/편집/조회), 이메일

## 5. 기능 범위

- **조회:** 현행 지도보기/목록보기 + 필터/검색(그대로 이전)
- **편집(웹):** 대리점 행 추가·수정·삭제, 주소 변경 시 즉시 지오코딩·검증
- **편집(엑셀):** 엑셀 업로드 → 파싱(SheetJS) → 미리보기(추가/수정/삭제 건수) → 검증 → 일괄 반영(전체교체/지점코드 기준 병합)
- **인증/권한:** 로그인, 역할별 보기/편집 분리
- **모바일:** 반응형/PWA로 폰 조회·간단 수정
- **(확장) 활동관리:** 대리점별 방문·상담 기록, 일정, 활동 대시보드/리포트

## 6. 단계별 로드맵 (위험 낮은 순)

- **0단계 — 데이터 분리(브리지):** HTML에서 데이터 분리, 지점코드 등 키 복원. (현행 기능 동일, 구조만 정리)
- **1단계 — DB+API 도입:** Supabase에 agencies 테이블 생성, 155건 이관, 좌표 저장. 지도 앱이 API에서 읽도록 전환.
- **2단계 — 편집 기능:** 인증(로그인) + 웹 CRUD + 엑셀 업로드(미리보기·검증·병합).
- **3단계 — 모바일 대응:** 반응형/PWA, 도메인 배포(+Naver 도메인 등록).
- **4단계 — 활동 관리 확장:** activities/users 테이블, 활동 입력·이력·리포트 → "활동 관리 시스템"화.

각 단계는 독립적으로 가치가 있고, 현재 앱은 마이그레이션 중에도 계속 사용 가능.

## 7. 주요 결정·리스크 체크리스트

- **비용:** Supabase/Firebase/호스팅 무료 티어로 시작 충분(155건·소규모). 사용량 증가 시 유료 검토.
- **보안/PII:** 연락처·주소 포함 → 공개 JSON 금지, 인증+RLS 필수. 권한 역할 정의 필요.
- **Naver 키/도메인:** 클라우드 배포 시 배포 도메인을 NCP Web 서비스 URL에 등록(미등록 시 지도 인증 실패). Geocoding API도 활성 유지.
- **데이터 정합성:** 지점코드 고유키, 업서트 규칙(전체교체 vs 병합), 변경 미리보기·백업.
- **지오코딩 실패:** 생성/수정 시 실패 건 플래그 + 관리 화면에서 수정 유도(현재 주소 오류 4건 사례처럼).
- **오프라인/현장:** 모바일 현장 사용 시 네트워크 의존. 필요 시 PWA 캐시 고려.
